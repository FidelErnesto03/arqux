# BLP-001: Ejecutar auditoría ejecutable de ArqUX siguiendo protocolo ARQUX_EXECUTABLE_AUDIT_PROTOCOL.md y dejar resultados en docs/
<!-- /BLP:TITLE -->

---

<!-- BLP:1 -->
## §1: Planteamiento del Problema

## Problema
Necesitamos ejecutar una auditoría ejecutable de ArqUX para determinar su estado de preparación para piloto, siguiendo el protocolo definido en `ARQUX_EXECUTABLE_AUDIT_PROTOCOL.md`.

## Evidencia
- El protocolo `ARQUX_EXECUTABLE_AUDIT_PROTOCOL.md` existe y define el proceso de auditoría ejecutable.
- Se ha identificado que ArqUX necesita una evaluación para adopción empresarial en modo piloto.

## Impacto de no resolverlo
Sin una auditoría ejecutable, no podemos determinar objetivamente si ArqUX cumple con los requisitos de calidad, seguridad y documentación necesarios para un piloto seguro, lo que podría llevar a riesgos en producción y pérdida de confianza en el framework.
<!-- /BLP:1 -->

<!-- BLP:2 -->
## §2: Objetivo

Ejecutar una auditoría ejecutable determinista de ArqUX siguiendo el protocolo ARQUX_EXECUTABLE_AUDIT_PROTOCOL.md y generar un informe de resultados en docs/ con formato ARQUX_AUDIT_REPORT_<timestamp>_UTC.md, sin modificar el código fuente.
<!-- /BLP:2 -->

- [ ] Verificar que el agente Heimdall tiene rol de auditor asignado

## §4: Principio Rector

**Principio Rector:** Seguir el protocolo ARQUX_EXECUTABLE_AUDIT_PROTOCOL.md al pie de la letra, utilizando únicamente los handlers MCP y la CLI de Arqux para recopilar evidencia, sin modificar el código fuente durante la auditoría.

**Evidencia del problema:** El protocolo define el proceso de auditoría ejecutable y se requiere cumplimiento estandarizado para obtener resultados verificables.

**Impacto si se viola:** Desviarse del protocolo produciría evidencia no válida o incompleta, invalidando la auditoría y potencialmente introduciendo sesgos o errores en la evaluación de preparación para piloto.

## §6: Alcance y Exclusiones

**Dentro del alcance:**
- Instalación y empaquetado (pip install, arqux init, copia de skills/workflows/)
- Inicialización en tiempo de ejecución (arqux handlers, versiones consistentes)
- Consistencia de handlers y superficie (conteo, registro, badges)
- Permisos y seguridad (modelo de roles, HMAC, pruebas de permisos, cobertura)
- Evidencia y auditabilidad (handlers de evidencia, detección de manipulación, hash de integridad)
- Tests y CI (pytest, cobertura, pruebas críticas, ausencia de CI workflow)
- Documentación para piloto (SECURITY.md, HANDLERS.md, PERMISSIONS.md, PILOT_MODE.md, etc.)
- Anti-patrones en VCS (archivos .bak, estado de .arqux/ en raíz)
- Ciclo de vida de blueprints (máquina de estados, transiciones, gates, role enforcement)
- Score y justificación (dimensiones de auditoría, cálculo de puntuación)
- Lista de mejoras P0/P1/P2 (acciones específicas para alcanzar PILOTO 100%)

**Fuera del alcance (excluido explícitamente):**
- Modificación directa de cualquier archivo bajo src/arqux/ o cualquier archivo .cortex
- Ejecución de pruebas unitarias o de integración como evidencia principal (aunque se pueden correr para obtener evidencia de estado)
- Creación o modificación de blueprints distintos a BLP-001 durante esta auditoría
- Evaluación de rendimiento o benchmarks no especificados en el protocolo
- Revisión de licencias o aspectos legales más allá de la presencia de LICENSE
- Análisis de dependencias externas fuera del alcance de instalabilidad básica

## §11: Procedimiento de Trabajo

1. Confirmar que estamos en el ciclo CYCLE-02 y que el agente Heimdall tiene rol de auditor.
2. Ejecutar `arqux handlers` para obtener la lista de handlers y contar.
3. Ejecutar `git rev-parse HEAD` y `git status` para obtener commit y estado de árbol de trabajo.
4. Ejecutar `arqux --version` y verificar consistencia con pyproject.toml, constants, etc.
5. Ejecutar `arqux init` en un directorio temporal para verificar empaquetado de skills/workflows y plantillas.
6. Verificar instalación de AGENTS.md mediante comparación de hash.
7. Listar handlers y verificar módulos (blueprint, cortex, cycle, evidence, identity, project, protocol, session, setup, skill, task, workspace).
8. Ejecutar pruebas de permisos (`pytest tests/test_permissions.py`) y observar fallos.
9. Ejecutar suite completa de pruebas con `pytest -q` y generar reporte de cobertura con `pytest --cov=arqux --cov-report=term-missing`.
10. Ejecutar pruebas específicas de blueprint learning, learn trigger, protocol y rename.
11. Verificar la presencia de documentos obligatorios (SECURITY.md, HANDLERS.md, PERMISSIONS.md, PILOT_MODE.md).
12. Buscar archivos .bak en el repositorio y en .arqux/.
13. Ejecutar `arqux sync_brain` o observar el output de `arqux init` para detectar el error silencioso de sync_brain.
14. Compilar toda la evidencia en un documento Markdown siguiendo la estructura del protocolo.
15. Calcular el score según las dimensiones definidas en el protocolo (installation_and_packaging, runtime_initialization, handlers_and_surface_consistency, permissions_and_security, evidence_and_auditability, tests_and_ci, documentation_for_pilot).
16. Escribir el informe final en docs/ARQUX_AUDIT_REPORT_<timestamp>_UTC.md.
17. Notificar al Arquitecto que la auditoría está completa y el informe está listo para revisión.

## §15: Riesgos

| ID | Riesgo | Impacto | Mitigación |
|---|---|---|---|
| R-01 | El proceso de auditoría podría intentar mutar archivos .cortex sin usar los handlers adecuados | Alto — viola la regla de no modificar código fuente | Usar exclusivamente handlers MCP; establecer ARQUX_STRICT_ROLES=1 |
| R-02 | La recolección de evidencia podría depender de comandos externos no aprobados | Medio — evidencia no válida según protocolo | Limitarse a handlers MCP documentados y CLI de Arqux |
| R-03 | La generación del informe podría tomar más tiempo de lo esperado | Bajo — retraso en la entrega | Ejecutar pruebas con timeout; priorizar secciones críticas |
| R-04 | El entorno de prueba no está aislado y podría afectar otros ciclos/proyectos | Medio — contaminación de datos | Usar directorio temporal para arqux init; no mutar archivos existentes |
| R-05 | Documentos obligatorios ausentes (SECURITY.md, etc.) podrían llevar a evaluación subjetiva | Medio — score impreciso | Documentar ausencias explícitamente; no inferir conformidad |
<!-- /BLP:15 -->

## §16: Regla de Bloqueo

Si se intenta modificar cualquier archivo bajo src/arqux/ o cualquier archivo .cortex directamente, o se utiliza un medio de obtención de evidencia que no sea un handler MCP o la CLI de Arqux (excepto para escritura del informe), o no se puede generar el informe de auditoría en docs/ debido a permisos de escritura o falta de espacio, o el protocolo ARQUX_EXECUTABLE_AUDIT_PROTOCOL.md no está disponible o es ilegible, entonces la auditoría se considera bloqueada y debe pasarse a estado bloqueado.
