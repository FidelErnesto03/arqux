## TITLE
P1 — cortex.ref + cortex.format: Utilidades CORTEX de consulta y transformación
<!-- /BLP:TITLE -->

<!-- BLP:1 -->
## §1: Planteamiento del Problema

El agente necesita conocer la sintaxis exacta de los sigils CORTEX para escribir contenido nativo (canal I). Hoy no hay forma de consultar "¿cómo se escribe un LNG?" sin leer la documentación. Tampoco hay forma de validar que un bloque CORTEX esté bien formado antes de enviarlo a un handler.

Además, el agente produce contenido en CORTEX (denso, para la máquina) pero el Arquitecto consume HCORTEX (expandido, para humanos). No hay un puente automático entre ambos formatos.

**Evidencia:**
- Sesiones donde el agente escribe sigils con sintaxis incorrecta
- El agente no tiene un mecanismo de auto-consulta para recordar sintaxis de sigils
- La transformación CORTEX↔HCORTEX se hace manualmente

**Impacto de no resolverlo:**
El canal I (content CORTEX en handlers) no puede funcionar si el agente no tiene herramientas para escribir y validar CORTEX correctamente.
<!-- /BLP:1 -->

<!-- BLP:2 -->
## §2: Objetivo

Crear dos handlers de utilidad CORTEX que le den al agente la capacidad de:

1. **cortex.ref(sigil)** — consultar en tiempo real la sintaxis canónica de cualquier sigil CORTEX (IDN, FCS, OBJ, LNG, RSK, KNW, SES, BLP, WRK, PULSE, etc.) con ejemplos de uso.
2. **cortex.format(content, target_format)** — transformar contenido entre formatos CORTEX (denso, para la máquina) y HCORTEX (expandido, para humanos), y viceversa.

Ambos handlers escriben PULSE en brain.cortex §7 para trazabilidad.
<!-- /BLP:2 -->

<!-- BLP:4 -->
## §4: Principio / Base de Diseño

1. **cortex.ref** es puramente declarativa — devuelve la sintaxis sin side effects. Basada en el esquema de sigils del CODEC-CORTEX (codec_cortex.py). Sin conexión a red.
2. **cortex.format** es transformativa — nunca modifica el original. Acepta: 'hcortex' (CORTEX → HCORTEX), 'cortex' (HCORTEX → CORTEX), 'puml' (CORTEX → PlantUML).
3. Ambos handlers escriben PULSE (`$7/PULSE:ref` y `$7/PULSE:format`) para que quede registro en brain.cortex.
4. Los errores son descriptivos: sigil inválido devuelve "Sigil no reconocido: X. Sigils válidos: [...]".
5. El canal I se mantiene para que la entrada/salida pueda ser CORTEX nativa cuando se invoque desde otro handler.
<!-- /BLP:4 -->

## §8: Diseño Técnico

### Estructura de handlers

**cortex.ref**
```
Handler: cortex.ref
Input:  sigil (str) — nombre del sigil CORTEX
Output: dict { sigil, name, p_level, sintaxis, expansion_type, ejemplo }
IO Type: native
PULSE:  $7/PULSE:ref — { sigil, ts }
```

Implementación: leer de `CANONICAL_SIGILS` en codec-cortex (`errors.py:331`) — tabla inmutable con 25 sigils canónicos. No hardcodear tabla separada. Para cada sigil, derivar sintaxis de su `expansion_type` (attrs, cuerpo, bloque, relación, attrs-pos). Ejemplos tomados del estándar CODEC-CORTEX.

**cortex.format**
```
Handler: cortex.format
Input:  content (str), target_format (str: "cortex"|"hcortex"|"puml")
Output: dict { original_content, transformed_content, format, warnings[] }
IO Type: native
PULSE:  $7/PULSE:format — { char_count_in, char_count_out, format, ts }
```

Implementación — delegar en CODEC-CORTEX existente, NO reimplementar:
- `cortex` target → codec-cortex `write_cortex()` (re-serialización canónica)
- `hcortex` target → codec-cortex `render_hcortex()` (v2/view_renderer.py)
- `puml` target → codec-cortex `render_hcortex_diagram()` o extraer bloques PUML del AST

Si el contenido de entrada no es CORTEX válido, intentar parsear con `parse_cortex()`. Si falla, pasar por `parse_hcortex()` + `encode_cortex_from_ast()` para detectar si es HCORTEX de entrada.

### Ubicación
`src/arqux/handlers/cortex_utils.py` — nuevo archivo. Importa de CODEC-CORTEX core (errors.py, writer.py, v2/view_renderer.py). Sin dependencias circulares.

## §9: Diseño Operacional

1. Registrar ambos handlers en REGISTRY con canal I nativo
2. El endpoint CLI se mapea como:
   - `cortex.ref` → `cortex.ref --sigil LNG`
   - `cortex.format` → `cortex.format --target hcortex`
3. El handler responde en formato nativo (dict) cuando se llama desde otro handler, y como markdown cuando se llama desde CLI
4. Los errores de sigil no encontrado retornan código 404 CORTEX
5. Sin autenticación adicional (misma sesión)
