# BLP-042: Establecer política nativa de uso de CODEC-CORTEX v0.5.0 y su CLI como único medio de escritura de archivos .cortex en ArqUX. Inventariar todos los handlers que escriben .cortex, definir esquema de integración, y eliminar los bypasses que producen entries incompletos (E032, E034) y archivos write-locked.
<!-- /BLP:TITLE -->

<!-- BLP:1 -->
## §1: Planteamiento del Problema

ArqUX escribe archivos `.cortex` sin pasar por CODEC-CORTEX, el motor que define el protocolo. Esto produce tres consecuencias críticas:

**A. Entries incompletos.** CODEC-CORTEX exige campos requeridos por sigilo (`LNG` requiere `prevention`, `OBJ` requiere `success`, todo entry requiere `name`). Los handlers de ArqUX escriben LNG con solo `type`, `cause` y `lesson` — sin `prevention` ni `name`. El validador emite E032_CRITICAL_SIGIL_INCOMPLETE.

**B. Bypass activo.** El handler `identity.record()` (cortex.py L224-240) tiene un fallback explícito: cuando `crud_add()` falla porque CODEC-CORTEX rechaza entries incompletos, ignora el error y escribe directamente con `write_text()`. Esto empeora el archivo en cada operación.

**C. Archivos write-locked.** `cortex format` y `cortex verify --strict` no pueden re-serializar los archivos dañados.

**D. El problema es sistémico, no aislado.** No solo alfred.cortex está afectado. El inventario completo revela errores en todas las identidades y varios archivos de gobierno:

## §2: Objetivo

Establecer como política nativa de ArqUX que **toda escritura de archivos .cortex debe realizarse exclusivamente a través de CODEC-CORTEX** (vía su CLI `cortex` o su API Python `cortex.core.writer.write_cortex()`), eliminando todo bypass directo. ArqUX debe mantener compatibilidad evolutiva con CODEC-CORTEX — no se fija a una versión específica, sino que usa la instalada. Esto incluye:

1. Reparar los 4 handlers identificados que escriben sin CODEC-CORTEX
2. Eliminar los fallbacks que hacen string append cuando CODEC-CORTEX rechaza una escritura
3. Reparar `alfred.cortex` y otros archivos con E032 acumulados
4. Verificar que ningún handler nuevo pueda evadir CODEC-CORTEX

## §4: Principio Rector

**ArqUX no escribe .cortex. ArqUX le pide a CODEC-CORTEX que lo haga.**

CODEC-CORTEX es el único motor de persistencia autorizado para archivos `.cortex`. ArqUX construye modelos de datos (frontmatter, entries) y delega la serialización, validación y escritura a CODEC-CORTEX.

**Regla fundamental:** Cualquier handler que necesite escribir o modificar un archivo `.cortex` DEBE usar `cortex.core.writer.write_cortex()` o el CLI `cortex add/update/delete`. NO se permite:

- `open().write()` directo
- `Path.write_text()` sobre .cortex
- String append con f-strings
- Fallback que bypass ee E032/E034
- `_render_cortex()` (el builder de strings obsoleto)

**Excepción única:** El handler `cortex.write` (MCP) recibe texto CORTEX y lo escribe — pero incluso él debe pasar por `cortex.core.writer.write_cortex()` para validar.

**Compatibilidad evolutiva:** ArqUX no fija versiones de CODEC-CORTEX. Usa la versión instalada. La integración debe funcionar con cualquier versión que exponga las APIs canónicas (`write_cortex`, `parse_cortex`, `validate`).
<!-- /BLP:4 -->

<!-- BLP:5 -->
## §5: Contexto

La arquitectura actual de ArqUX tiene 3 capas de interacción con CODEC-CORTEX:

**Capa 1 — CODEC-CORTEX (librería externa)**
- `cortex.core.writer.write_cortex()` — escribe documentos desde AST
- `cortex.core.parser.parse_cortex()` — parsea texto a AST
- CLI `cortex add/update/delete/format` — operaciones CRUD y formateo
- `cortex.core.validator.validate()` — verifica cumplimiento de contratos
- NO se modifica — es dependencia externa

**Capa 2 — ArqUX state.py (capa de integración)**
- `crud_add()` — intenta usar `atomic_write_cortex()`, pero falla a bypass cuando hay non-bypassable errors
- `write_cortex_pair()` — escribe pares .cortex/.md, usa `_build_doc()` + CODEC-CORTEX para governance, `_render_cortex()` (string legacy) para otros
- `_HAS_CODEC_CORTEX` flag — determina si CODEC-CORTEX está disponible

**Capa 3 — Handlers (punto de entrada MCP)**
- `record_lesson_handler()` — bypass: string append directo cuando crud_add falla
- `_record_to_brain()` — bypass: string append directo en brain PULSE
- `ensure_container()` — bypass: write_text directo
- `cortex.write_handler` — ✅ usa CODEC-CORTEX
- `cortex.entry.add/update/delete` — ✅ usan CODEC-CORTEX

**El problema está en la Capa 2 (fallback) y Capa 3 (bypass directo).**

```
Flujo de escritura actual:

Handler MCP → crud_add() → atomic_write_cortex() → ERROR E032
                              ↓
                        fallback: non_bypassable
                              ↓
                   Handler escribe string append directo
                   (sin prevention, sin name, sin validacion)
```

La Capa 2 ofrece un camino de escape cuando CODEC-CORTEX rechaza una escritura. La Capa 3 lo toma y escribe igual, empeorando el archivo cada vez.

## §7: Reglas Obligatorias

1. **CODEC-CORTEX es el único writer.** Ningún handler puede llamar `write_text()`, `open().write()` o similar sobre archivos `.cortex`. Toda escritura DEBE pasar por `cortex.core.writer.write_cortex()`.

2. **Sin excepción por E032/E034.** Si CODEC-CORTEX rechaza una escritura por campos incompletos, el handler NO DEBE hacer bypass. Debe reportar el error y detenerse. La reparación se hace vía `cortex update` o `cortex recover`, no vía string append.

3. **Los LNG requieren 4 campos.** Todo LNG entry debe incluir `type`, `cause`, `lesson`, `prevention`. El handler `identity.record()` debe recibir `prevention` como parámetro requerido.

4. **Los archivos .cortex reparados deben pasar `cortex verify --strict`.** Antes de cerrar este BLP, todos los .cortex del workspace deben ser verificables sin errores bloqueantes.

5. **Declarar campos en $0.** Todo archivo .cortex gestionado por ArqUX debe declarar los campos requeridos de sus sigilos en el glosario $0, usando la columna Fields en la pipe-table.

6. **`crud_add()` no debe retornar non_bypassable.** Si CODEC-CORTEX rechaza, el error se propaga al handler y de ahí al usuario. Nunca se escribe igual.

7. **`mcp-handlers.skill.md` debe reflejar estas reglas.** El skill de handlers debe incluir los nuevos AXM de política de escritura CODEC-CORTEX, y el HDL de `identity.record` debe declarar `prevention` como parámetro en su firma.

## §8: Diseño Técnico

**8.1 Esquema de integración CODEC-CORTEX en ArqUX**

Cada handler que escribe .cortex debe seguir este contrato:

```python
# ✅ ÚNICA FORMA AUTORIZADA
from cortex.core.ast import CortexDocument, Section, Entry, SigilDef
from cortex.core.writer import write_cortex
from cortex.core.parser import parse_cortex

# 1. Construir o modificar el documento vía AST
doc = parse_cortex(text)           # leer existente
doc.sections[1].entries.append(    # añadir entry
    Entry(sigil="LNG", name="...", value={...})
)

# 2. Validar
from cortex.core.validator import validate
errors = validate(doc)
if any(e["severity"] == "error" for e in errors):
    raise RuntimeError(f"CODEC-CORTEX validation failed: {errors}")

# 3. Escribir
write_cortex(doc, path)
```

```python
def ensure_container(self) -> None:
    if self.path.exists():
        return
    self.path.parent.mkdir(parents=True, exist_ok=True)
    # Usar cortex.core.writer en vez de write_text()
    from cortex.core.ast import CortexDocument, Section, Entry, SigilDef
    doc = CortexDocument()
    sec0 = Section(id="$0", title="")
    doc.sections.append(sec0)
    doc.glossary.add_sigil(SigilDef(sigil="LNG", ...))
    # Añadir $0.1 ARQX:artifact si aplica
    sec01 = Section(id="$0.1", title="ARQUX METADATA")
    sec01.entries.append(Entry(sigil="ARQX", name="artifact", value={...}))
    doc.sections.append(sec01)
    sec1 = Section(id="$1", title="LESSONS")
    doc.sections.append(sec1)
    write_cortex(doc, str(self.path))
```

**8.4 Reparación de _record_to_brain (blueprint.py)**

```python
def _record_to_brain(root, bp_id, outcome, evidence):
    brain_path = root / ".arqux" / "brain.cortex"
    text = brain_path.read_text()
    doc = parse_cortex(text)
    pulse_sec = next(s for s in doc.sections if s.id == "$12" or "PULSE" in s.title)
    pulse_sec.entries.append(
        Entry(sigil="AUD", name=f"{bp_id}_{outcome}",
              value={"kind": "blueprint", "evidence": evidence, "date": now()})
    )
    write_cortex(doc, str(brain_path))
```

**8.5 Protocolo de migración de archivos existentes (backup → reescritura)**

```python
from pathlib import Path
from cortex.core.ast import CortexDocument, Section, Entry, SigilDef

def migrate_cortex_file(path: Path, *, dry_run: bool = False) -> bool:
    """Migra un .cortex dañado: backup .bck → reescritura limpia.
    
    Retorna True si migró, False si no tenía errores o no aplica.
    """
    if not path.exists():
        return False
    
    # 1. Verificar si tiene errores
    from cortex.core.parser import parse_cortex
    from cortex.core.validator import validate
    
    text = path.read_text(encoding="utf-8")
    try:
        doc = parse_cortex(text)
        errors = validate(doc)
        has_blocking = any(e["severity"] == "error" for e in errors 
                          if e["code"] in ("E032", "E034", "E008"))
        if not has_blocking:
            return False  # no necesita migración
    except Exception:
        has_blocking = True  # ni siquiera se puede parsear
    
    # 2. Backup
    backup = path.with_suffix(".cortex.bck")
    if not backup.exists():
        path.rename(backup)
    
    if dry_run:
        return True
    
    # 3. Leer datos del backup
    raw = backup.read_text(encoding="utf-8")
    
    # 4. Construir documento limpio desde los datos extraídos
    # (cada tipo de archivo sabe cómo construir su propio doc)
    new_doc = _build_clean_document(path.stem, raw)
    
    # 5. Escribir nuevo archivo
    write_cortex(new_doc, str(path))
    
    # 6. Verificar
    final = parse_cortex(path.read_text(encoding="utf-8"))
    final_errors = validate(final)
    assert not any(e["severity"] == "error" for e in final_errors), \
        f"Migración produjo errores: {final_errors}"
    
    return True
```

**8.6 Glosario $0 con campos declarados**

Cada archivo .cortex de ArqUX debe declarar en su $0 los campos esperados de cada sigilo:

```
# -- $0: ARQUX GLOSSARY --
# Sigil | Name | Type | Risk | Layer | Description | Fields
# LNG   | lesson     | attrs      | M | Episodic       | Behavioral lesson     | type, cause, lesson, prevention, name
# OBJ   | objective  | attrs      | H | Working        | Active goal           | goal, status, success, survive, name
# AUD   | audit      | attrs      | M | Prefrontal     | Verification record   | event, evidence, result, date, name
```
